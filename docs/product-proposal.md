## 一、背景与问题

### 1.1 现状痛点

作为出海制造集团，公司财务数据具有三个天然复杂度，导致"取一个数"的成本远高于外界想象：

1. **多主体**：集团含母公司、多家境内子公司、香港/马来西亚/美国境外主体，每家一套报表
2. **多口径**：同一"净利润"存在单体 / 合并、账面 / 审定、含少数股东损益与否等多种合法口径
3. **多币种**：境外主体存在本币与人民币双版本报表

由此产生的实际问题：

- **取数慢**：业务和管理层想知道"某子公司某月毛利率"，需要找到对应文件、打开正确的表、定位正确的口径，或直接问财务同事等回复，单次平均耗时 15–30 分钟
- **口径乱**：不同人对同一指标的默认理解不同，跨部门沟通反复确认，甚至出现汇报材料口径不一致
- **知识散**：财务制度、操作规范散落在多个 Word / PDF 中，新人和业务方查一条规定要翻半天，人员流动时经验随人流失
- **财务负担重**：财务同事大量时间被"帮别人查数 / 解释口径"占用，无法聚焦分析性工作

### 1.2 问题的本质

这不是"报表不够多"的问题，而是**财务数据与知识的"最后一公里"没有打通**——数据都在，但获取门槛高、口径不透明、无法自助。这正是自然语言问答（RAG + NL2SQL）技术能解决的典型场景。

## 二、产品方案

### 2.1 核心设计：双通道

用户用一句话提问，系统自动判断意图并走不同通道：

- **财务数据问题**（"星辰集团合并口径净利润多少"）→ NL2SQL 通道：自动选表、生成 SQL、返回数据 + 原表下载
- **规章制度问题**（"应付月结后怎么改数据"）→ 文本 RAG 通道：混合召回相关制度段落，答案强制附来源引用

### 2.2 已实现的关键能力（技术验证已通过）

| 能力 | 说明 | 解决的痛点 |
|------|------|-----------|
| 多公司多口径识别 | 按问题关键词预筛表；未指明口径时单体/合并双口径对照返回 | 口径乱 |
| 多币种处理 | 境外主体本币/人民币双版本同时返回并标注 | 多币种 |
| 混合召回 | BM25（精确术语）+ 向量（语义近似）+ RRF 融合 | 制度查得准 |
| 公司名同音纠错 | 拼音匹配提示候选 + 用户确认，绝不静默替换 | 查错公司比查不到更危险 |
| 评估门禁 | 黄金测试集，改动前后必跑，分数不降才合入 | 保证准确率不退化 |
| 安全治理雏形 | SQL 仅允许 SELECT、敏感字段脱敏、查询审计日志 | 数据安全 |

### 2.3 演示

系统已可完整演示上述全部场景（脱敏版本已开源，含架构图与演示动图）。这意味着**产品可行性已经得到验证，本方案讨论的是"如何从可用原型变成可稳定供全员使用的生产系统"**，而非从零开发。

## 三、目标用户与使用场景

### 3.1 用户分层（约 50+ 人）

| 用户群 | 规模（示意） | 核心诉求 | 权限范围 |
|--------|------------|---------|---------|
| 财务人员 | ~15 人 | 快速核对口径、减少被打断 | 全量数据 |
| 业务分析 / BP | ~15 人 | 自助取经营数据、做分析 | 本条线 / 本区域数据 |
| 部门负责人 / 管理层 | ~10 人 | 快速了解经营指标 | 汇总口径为主 |
| 其他相关岗位 | ~10 人 | 查制度、查流程 | 制度文档 + 有限数据 |

> 权限分层是 50 人规模的**硬约束**：不同用户能看的数据范围必须不同（如区域 BP 只能看本区域、普通岗位看不到薪酬类科目），详见第五章。

### 3.2 典型场景

1. 业务经理问"我负责的产品线本季度毛利率" → 秒级返回，不用等财务
2. 管理层问"集团合并口径营收同比" → 直接拿到数 + 口径标注
3. 新员工问"差旅费报销标准" → 返回制度原文 + 出处
4. 财务问"某子公司某科目余额" → 自助核对，省去翻表

## 四、数据合规评估（核心章节）

> 财务数据是公司最敏感的资产之一。任何 AI 系统上线前，**数据流向的合规性是"一票否决"项**——这一章不通过，其余章节均无意义。

### 4.1 数据流向分析

系统运行时，用户问题和检索到的财务数据会作为上下文发送给大模型生成答案。**关键风险点在于：财务数据是否会离开公司可控范围。**

| 数据类型 | 敏感级别 | 是否可外发 |
|---------|---------|-----------|
| 规章制度文档 | 内部 | 视内容评估，多数可控 |
| 单个科目数值 | 机密 | 谨慎 |
| 完整报表 / 经营指标 | 机密 / 绝密 | **不可外发第三方** |

### 4.2 两条技术路线对比

| 维度 | 路线 A：第三方 API（DeepSeek 等） | 路线 B：私有化部署开源模型 |
|------|-----------------------------------|---------------------------|
| 数据安全 | 数据发送至外部服务商，存在合规风险 | **数据全程不出内网，合规最优** |
| 一次性成本 | 低（无需硬件） | 中（需 GPU 服务器，约 3–5 万或按需租用） |
| 使用成本 | 按调用量付费，随规模上升 | 硬件折旧 + 电费，边际成本近零 |
| 效果 | 顶尖闭源模型，效果最好 | 开源模型（Qwen2.5 / DeepSeek 开源版）已足够胜任本场景 |
| 运维 | 几乎免运维 | 需内部维护 GPU 服务器与模型 |
| 可控性 | 依赖服务商稳定性与政策 | 完全自主可控 |

### 4.3 推荐方案

**推荐路线 B（私有化部署）作为生产环境的数据通道方案**，理由：

1. 公司数据规模（50+ 人高频使用）下，路线 A 的按量付费成本会持续上升，路线 B 边际成本近零，长期更划算
2. 财务完整报表属机密/绝密级，**私有化是唯一能彻底消除数据外发风险的方案**
3. 本场景对模型能力要求不极端（选表、生成 SQL、总结制度），开源模型已验证可胜任
4. 公司已表示可能支持私有化部署，具备落地条件

**分阶段策略**：试点期可先用"第三方 API + 严格脱敏"快速验证价值（仅处理制度类非敏感数据），正式上线切换为私有化部署处理全部数据通道。

### 4.4 无论哪条路线都要做的合规基线

- **数据分级**：建立财务数据敏感度分级表，系统按级别控制处理方式
- **脱敏**：身份证 / 银行卡 / 个人信息自动脱敏（已实现雏形）
- **审计**：全量查询日志，可追溯"谁、何时、查了什么"
- **最小权限**：用户只能访问职责范围内的数据

## 五、上线改造项（从原型到生产）

当前原型为满足单人验证而设计，供 50+ 人使用需在四个方面升级。这是本方案申请资源的主要投向。

| 模块 | 原型现状 | 生产目标 | 优先级 |
|------|---------|---------|--------|
| **大模型** | 第三方 API | 私有化部署开源模型（见第四章） | P0 |
| **账号与权限** | 全站单一口令 | 个人账号（对接公司 SSO/LDAP）+ 行级/列级数据权限 | P0 |
| **数据库** | SQLite（单写锁） | PostgreSQL（支持并发） | P0 |
| **Web 架构** | Streamlit 单进程 | 支持 50+ 并发（Streamlit 多副本 或 迁移 FastAPI + 前端） | P1 |
| **部署** | 本地手动启动 | Docker 化 + 内网服务器 + nginx 反向代理（模板已备） | P1 |
| **数据维护** | 手动重建索引 | 明确的月度数据更新 SOP + 责任人 | P1 |
| **监控备份** | 无 | 日志监控 + 异常告警 + 数据库每日备份并演练恢复 | P1 |

> 说明：账号权限与数据库并发是 50 人规模从"能用"到"能稳定供多人用"的分水岭，列为 P0。

## 六、投入产出测算（ROI）

> 以下为测算模型与示意数值，正式提交前请用公司实际数据替换方括号内数字。

### 6.1 收益（人力节省）

```
活跃用户数            ≈ [30] 人（50 人中高频取数者）
人均每周取数次数      ≈ [5] 次
单次节省时间          ≈ [20] 分钟（从 ~25 分钟降到 ~2 分钟）
─────────────────────────────────────────────
每周节省             = 30 × 5 × 20 = 3,000 分钟 ≈ 50 小时
每月节省             ≈ 200 小时
折算人力成本（时薪 [80] 元） ≈ 200 × 80 = 1.6 万元/月
年化节省             ≈ [19] 万元
```

### 6.2 成本

| 项目 | 一次性 | 年度经常性 |
|------|-------|-----------|
| GPU 服务器（私有化部署） | [3–5] 万元 | 折旧 + 电费 [1] 万元 |
| 开发改造人力（账号/权限/并发） | 折算 [2] 万元 | — |
| 运维人力 | — | [1–2] 万元 |
| **合计** | **~[5–7] 万元** | **~[2–3] 万元** |

### 6.3 结论

**年节省约 19 万元 vs 首年总投入约 8–10 万元，预计半年内回本，之后持续净收益。**

### 6.4 难以量化但真实的收益

- **口径统一**：减少因口径不一致导致的汇报错误（一次重大错误的代价可能远超系统成本）
- **知识沉淀**：制度与经验固化进系统，人员流动不丢失
- **决策提速**：管理层实时拿到经营数据，决策更快
- **财务解放**：财务同事从"人肉查询机"中解放，转向分析性工作

## 七、分阶段落地计划

| 阶段 | 时间 | 目标 | 范围 | 里程碑 |
|------|------|------|------|--------|
| **一、试点** | 1 个月 | 验证真实价值、收集反馈 | 财务部 + 2–3 名业务方 | 采纳率、纠错率、满意度达标 |
| **二、部门推广** | 2–3 个月 | 扩大使用、完善权限 | 整个财务条线 | 私有化部署上线、账号权限完备 |
| **三、全条线上线** | 持续 | 稳定运行、持续迭代 | 50+ 全部目标用户 | 监控备份完善、月度数据 SOP 运转 |

**灰度原则**：每阶段设明确的通过标准（准确率、采纳率），达标才进入下一阶段，风险分段可控。

## 八、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| **AI 答错数字进了汇报材料** | 高 | 每条答案标注数据来源与口径、提供原表下载、明确"结果需复核"使用规范、置信度提示 |
| 数据合规不通过 | 高（一票否决） | 私有化部署 + 数据分级 + 审计（第四章） |
| 用户不会用 / 不愿用 | 中 | 试点期贴身陪跑、一页纸操作指南、收集反馈快速迭代 |
| 数据更新不及时导致答案过时 | 中 | 建立月度数据更新 SOP + 明确责任人 |
| 并发性能不足 | 中 | 数据库升 PostgreSQL、高频查询缓存、压测后上线 |
| 权限泄露（越权看到数据） | 高 | 行级/列级权限在系统层强制，LLM 不接触越权数据 |

### 核心设计原则

> **财务场景下，给错一个数字比查不到更严重。** 因此系统的准确率策略是分层的：高频标准问题走可靠路径并附口径标注，长尾问题明确提示"需人工复核"，而非追求单一模型的整体准确率。这也是本产品区别于通用 ChatBI 的关键。
